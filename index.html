Foundations · how machine learning differs from software

Software follows rules. Machine learning learns from examples.

The old way · hand-written rules
if email contains "free money" → spam if email contains "viagra" → spam if email contains "you won" → spam … and 10,000 more rules a human typed, still missing every new trick.
The ML way · show examples

Show it 10,000 emails already marked spam / not-spam. The machine writes its own rule — it learns the vibe of spam, including tricks no human listed.

Nobody can list every spam phrase — but you can show examples
That flip — from writing rules to showing examples — is the entire foundation. Everything today is built on it.
Three ways to train a machine-learning model

The 3 methods to teach a model.

They differ by one thing: what you hand the model during training.

Method 1

Supervised

Give the model both the input and the correct output; it learns the mapping between them.

needs labelled data
Method 2

Unsupervised

Give the model only inputs, no labels; it discovers structure and patterns on its own.

no labels required
Method 3

Reinforcement

Give the model a reward signal, not answers; it acts, gets scored, and adjusts to maximise reward.

learns from a score
We'll define each precisely — input, output, and objective — then later see which method trains each stage of an LLM.
Method 1 of 3 · supervised learning

Give the model input + correct output. It learns the mapping.

the setup

We supply a dataset of labelled pairs — each input paired with its correct output. The model predicts, we compare to the label, and we adjust it to reduce the error. Repeat until it generalises to inputs it has never seen.

Method 3 of 3 · reinforcement learning

No answers — just a reward. The model learns by trial.

the setup

An agent takes an action in an environment and receives a reward (or penalty). It is never told the correct action — it explores, and adjusts its strategy to maximise total reward over time.

The "before times" · pre-2017

For decades, humans hand-wrote every language rule.

# armies of linguists, by hand: if word.endswith("ed"): tense = "past" sentiment = sum(GOOD_WORDS) - sum(BAD_WORDS) grammar = parse(CHOMSKY_RULES) # millions of them dictionary.lookup(word) or crash() # 💥 slang? irony?
  • Rule-based & dictionary-based. Give the computer a strict grammar textbook.
  • Brittle. One slang word, one bit of sarcasm, and the whole thing breaks.
  • Narrow. Only worked on short, clean sentences.
  • Doesn't scale. Every new language or domain = start completely over.
ELIZA · SHRDLU · early Google Translate · Clippy · autocorrect disasters

Text is free-flowing — what comes next depends on everything before.

text was a puzzle

To fill these blanks well, it has no choice but to learn…

To predict the last word of a mystery novel, you basically had to follow the whole plot. Forcing next-word prediction forces it to learn grammar, world facts, logic — and reasoning.
Self-supervision · the data labels itself

No humans required. The answer is already there.

"The capital of France is [ ? ]"
→ the label is the very next word
   already in the text: Paris
no annotators · no labeling cost · trillions of examples

Remember supervised learning's expensive human labels? Here's the magic: the label is just the next word that's already written.

You get the scale of unsupervised (nobody labeling) with the precision of supervised (there's a right answer every time). Nerds call it self-supervised learning — and it's why we can train on the whole internet.
This whole stage has a name:  PRE-TRAINING
So how do we actually train on raw text?

Turn "understand language" into a fill-in-the-blank game.

Take any sentence. Hide the next word. Make the model guess. Check against the real word. Nudge it. Repeat a few trillion times.

one sentence becomes many training examples
The predict → quick
The quick predict → brown
The quick brown predict → fox
That's it — that's the entire training objective for a base model. Sounds trivial. It is secretly everything.
A note on representation · tokens

Models don't see words. They see tokens.

Tokens are word-chunks — Lego bricks of text. It lets a model handle any word, even ones it's never seen, by building them from pieces. Type anything and watch it split. (For the rest of the talk I'll say "word" and "token" interchangeably.)

rule of thumb: ~750 words ≈ 1,000 tokens · 1 token ≈ ¾ word
This is why an LLM struggles to count the r's in "strawberry" — it never sees letters, only opaque chunks. It's also why APIs bill per token and have context limits in tokens. You're literally paying by the syllable.
Inference · sampling one token at a time

It outputs a probability over every possible next word.

Pick a word to feed it back in, and watch the sentence grow — exactly how the model generates, one token at a time.

Post-training begins · the stopping problem

"This thing knows language. Why don't we teach it to follow instructions — and stop?"

the idea

Build a dataset of instructions + their ideal answers, written by humans. End every answer with a special end-of-turn token — <|im_end|>. (Real models use this or <|eot_id|>.)

what changes

The model still just predicts the next token — but now it learns to predict <|im_end|> when the answer is done. The instant that token appears, the system forcibly stops it.

# a training example looks like: ### Instruction: Write a haiku about the sea. ### Response: Endless blue expanse / waves whisper ancient secrets / gulls trace the salt wind <|im_end|>
Stage 1 · supervised fine-tuning (SFT)

Instruction Fine-Tuning — show it good examples of being helpful.

Humans write thousands of high-quality "this is how a helpful assistant responds" pairs. We fine-tune the model to imitate them. Note: this is supervised learning again — labelled input→output pairs, exactly like Part 1.

Analogy: a brilliant new hire shadowing a great employee — "when a customer asks X, here's how we answer." The talent was already there; now it learns the format of the job.
This is why "act as a pirate 🏴‍☠️" works — it was trained to follow your instructions.

Instruction

"Explain photosynthesis to a 5-year-old."

Ideal response (human-written)

"Plants eat sunlight! Leaves catch light and turn it into food, breathing out the air we need. 🌱 <|im_end|>"

Result → an INSTRUCT model. Base weights, now able to answer and stop. This is the first post-training stage; watch the rail advance to 02.
The scaling limit of hand-written answers

Following instructions is great. But how do we teach vibe — and scale it?

How long? How formal? When to refuse? We can't learn this unsupervised — internet text is a chaotic mix (a meme isn't formal English). So… supervised again. But writing perfect answers by hand doesn't scale:

10 humans × 100 answers/day = 1,000/day
need 1,000,000 examples? → months
more humans · more time · 💸💸💸
the reframe that unlocked everything

Don't make humans write the perfect answer. Have the model generate 4 options — and ask humans only to pick the best one.

Writing → multiple-choice. A human can rank thousands a day. This sped everything up.
Step 2 · Preference ranking — be the human labeler

Don't write the answer. Rank the candidates.

The model generated four responses to one prompt. Click them best → worst. Thousands of these become a dataset of human preference.

Prompt: "My code throws a NullPointerException. Help?"
RLHF · Reinforcement Learning from Human Feedback

Stage 2 · reinforcement learning from human feedback.

model writes answers→ humans rank them→ train a reward model→ model practices to maximize reward
Cookie & hot stove, again. Good answer → reward. Rude or wrong → penalty. Do it a lot and behavior shapes up. The state = your question; the actions = the tokens it generates.

We can't hand-write the ideal answer to everything. But humans are great at comparing — "A is better than B." That comparison is all RL needs.

technical: reward model + PPO, anchored by a KL penalty so it doesn't drift into gibberish to game the score.

Chain-of-thought prompting

Letting the model show its working raises accuracy.

Chatbots would blurt an answer and get it wrong. Then people found: just ask it to show its work. Flip the switch.

Q: What's 17 × 24?
blurt it out
"408? 448? … 410."
— confident, wrong, no working
think step by step
click the button →
Analogy: a kid doing mental math vs. working it on paper. The paper isn't magic — it's room to think. Each written step becomes context for the next one.
From "please think" → "thinks on its own"

So they baked the scratchpad in.

Train the model so it always writes out its reasoning first — between special tags — then gives the final answer. You don't have to ask anymore. That little "thinking…" indicator? That's the scratchpad.

<reasoning> 17 × 24 = 17 × (25 − 1) = 425 − 17 = 408 </reasoning> The answer is 408. <|im_end|>
reasoning effort · low / medium / high

A dial for how long it stays thinking. Longer reasoning → better answers, but more tokens, more time, more cost. You're literally buying accuracy with compute.

These are the o-series (OpenAI o1 / o3), DeepSeek-R1, Claude's extended thinking — the first reasoning models (2024–25).
RLVR · RL from Verifiable Rewards

For math & code, you don't need a human judge — just check if it's right.

RLHF needed slow, expensive, sometimes-wrong human graders. But correctness in math and code is checkable by a machine. So the model attempts a hard problem thousands of ways — and the attempts that reach the verified-correct answer get rewarded.

No human grader. It trains itself tirelessly, in the dark. This killed "reward hacking" — you can't sweet-talk a unit test.
the verifier
attempt → answer: 408  ✓ +1 reward
attempt → answer: 412  ✗ 0 reward
attempt → answer: 408  ✓ +1 reward
code: does it run? do the tests pass?
Analogy: a kid practicing a Rubik's cube alone. Nobody grades each move — the cube is either solved or not. That self-check lets them grind and improve forever.
An emergent result · 2025

Reward only the correct answer, and human-like thinking emerges on its own.

"DeepSeek-R1 spontaneously learned to double-check itself and say 'wait, let me reconsider' mid-solution — nobody programmed that."

Backtracking, verifying, little "aha" moments — we didn't teach those behaviors. They emerged because they work. That's the difference between a model that sounds smart and one that actually works the problem.

Result → a REASONING model (rail stage 04) — OpenAI's o-series, DeepSeek-R1, Claude's extended thinking.
A limitation of static weights

A brilliant assistant… frozen in time.

Training pulls data from the web — but only up to a cutoff date. Ask about anything after it, and the model simply can't know.

You: Who won the 2026 NBA championship?
Model: My knowledge cutoff is January 2025, so I can't say.

Yet ask the same model in its app today and it answers — you see "🔍 Searching the web…" first. How did a frozen brain learn to look things up?

The ReAct framework · Reason + Act

Give the brain a pair of hands.

Hand the model a list of tools — each with a name, a description, and its inputs. Train it: "if you need one, emit a tool call between special tags."

# tools available: web_search(query) # look up the live web calculator(expr) # exact arithmetic # the model emits: <tool_code> web_search("2026 NBA champion") </tool_code>
the loop
1. model reasons about your request
2. emits <tool_code> instead of answering
3. we pause it, actually run the tool
4. feed the result back: "here's the output"
5. model continues, now informed
Reason about the request → Act by calling a tool → read the result → continue. That's ReAct. A consultant who could only talk now has a laptop, a terminal, and database access.
Multi-step RL with verifiable outcomes

Split a task into steps, act on each, and recover from failure.

Plan→ Act (tool)→ Observe result→ Adjust↺ Done ✓

Give it a real task — "fix this failing test." Let it loop: run a command, read the error, try again. Reward it only if the whole task succeeds.

why code is a superfood
  • Abundant — all of GitHub.
  • Verifiable — it runs or it doesn't; tests pass or fail. Perfect RLVR signal.
  • Transfers — training on code makes models better at general reasoning.
This teaches the model to handle failure — read an error, change approach, retry. That resilience is what lets Claude Code grind through a real bug instead of giving up after one try.

In practice · sourcing new training data

It comes from us. Every interaction is a training signal.

  • Every prompt you type into ChatGPT/Claude — stored, likely trains the next model.
  • "Which answer is better, A or B?" — that's you labeling preference data.
  • 👍 / 👎 buttons — direct reward signal.
  • Code autocomplete: accept = positive, reject = negative. Type "yes" = good signal; override its suggestion = a correction.
why the flywheel is worth billions

In June 2026, SpaceX/xAI agreed to buy Cursor for ~$60B. Not for the editor — for the real-world coding data from millions of developers, which exists nowhere else. (In 2025, OpenAI, Google & Cognition all fought over Windsurf for the same reason.)

This is why the best tools are free. You're not the customer — your interactions are the dataset.

In practice · Mixture-of-Experts architecture

Don't wake the whole company for every question. Route it.

Most modern models aren't one giant brain — they're many smaller experts + a router that wakes only the relevant 2 per word. Pick a question, watch who gets called.

Friend-group analogy: health question? Ask your doctor friend, not your manager. The shared experts (always on) are like your parents — they know everything about you, so they're in on every call.
